基本概念

并发&并行

并发：张三、李四、王五三人同时拉肚子去厕所，但是只有一个坑位。

并行：张三、李四、王五三人同时拉肚子去厕所，有三个坑位。

进程&线程&管程

进程：富士康（操作系统）有苹果生产线、华为生产线，他们互不影响…一个生产线代表一个进程

线程：一个生产线上有电池安装环节、芯片植入环节…一个环节代表一个线程。几个环节（多线程）抢着加工同一个手机，就造成了并发

管程：监视器，也就是我们常说的“锁”。可以理解为某生产线上一台手机，同一时间其只能被流水线上一个环节（线程）加工

用户线程&守护线程

一般不作特别配置，我们所创建的都是用户线程

用户线程：系统的工作线程，它会完成这个程序需要完成的业务操作。

守护线程：一种特殊线程为其他线程服务的，在后台默默完成一些系统性服务，如：垃圾回收线程。

用户线程就好比生产线上某一环节，比如手机塑化包装环节（某一用户线程），而守护线程则是该手机材料购进环节（该进程的守护线程）。如果有一天提倡低碳环保取消了塑化包装（某一用户线程执行完毕），材料引进则不再购进塑化包装所需材料（守护线程不再提供服务支持）。如果有一天其中一个手机生产线生产的手机总是爆炸，生产线停工面临整改（用户线程全部执行完毕），该手机生产线的材料引进则不再购进该手机生产所需全部材料（守护线程也会自动关闭）。

并发编程三大特性

注意，这三大特性不是因为多线程而拥有的，而是并发编程时一定会遇到的问题，要想保证多线程下程序运行没有问题，就必须要保证满足这三点。如此，健全的并发编程都具备这三个特点。

可见性

概念：

当多个线程访问同一个变量时，一个线程修改了这个变量的值，其他线程能够立即看得到修改的值。

问题描述：

解决方案：

  1. 将变量放在方法中，使用局部变量的方式（可重入代码）。

  2. 使用关键字保障数据的可见性，volatile、synchronized、final。

有序性

概念：

即程序执行的顺序按照代码的先后顺序执行

问题描述：

JVM在真正执行代码的时候，可能会发生指令重排序（处理器为了提高程序运行效率，可能会对输入代码进行优化，它不保证程序中各个语句的执行先后顺序同代码中的顺序一致，但是它会保证程序最终执行结果和代码顺序执行的结果是一致的）。处理器在进行重排序时是会考虑指令之间的数据依赖性，如果指令2必须用到指令1的结果，那么处理器会保证指令1在指令2之前执行。虽然重排序不会影响单个线程内程序执行的结果，但是多线程就无法保障了。

解决方案：

在 Java 中，可以使用 Synchronized 和 Volatile 来保证多线程之间操作的有序性。实现方式有所区别，Volatile 关键字会禁止指令重排。Synchronized 关键字保证同一时刻只允许一条线程操作。（单线程中任何操作都是有序的，一个线程看另一个线程的操作都是无序的）

原子性

概念：

一个或者多个操作作为一个整体，要么全部执行，要么都不执行，并且操作在执行过程中不会被线程调度机制打断；而且这种操作一旦开始，就一直运行到结束，中间不会有任何上下文切换。

问题描述：

账户转账问题就是个典型的原子性问题，不多赘述

解决方案：

在 Java 中，为了保证原子性，提供了两个高级的字节码指令 Monitorenter 和 Monitorexit。这两个字节码，在 Java 中对应的关键字就是 Synchronized。

锁概念

悲观锁&乐观锁

悲观锁

每次去拿数据时都会认为数据会被别人修改（悲观，总是假设坏的情况），所以读数据时就开始上锁，如此别人想在此期间拿到数据就会阻塞，直到其获得锁。典型的有synchronized、ReentrantLock

乐观锁

每次去拿数据时都会认为数据不会被别人修改（乐观，总是假设好的情况），所以每次拿数据时都不上锁，但是在更新的时候会判断一下在此期间别人有没有去更新这个数据（使用版本号机制和CAS算法实现），适用于多读的应用类型，这样可以提高吞吐量。在JUC包下的java.util.concurrent.atomic包中的原子变量类就是使用了乐观锁的一种实现方式CAS实现的。

悲观锁&乐观锁实现

悲观锁使用synchronized、ReentrantLock即可。

乐观锁实现方式一：版本号

在数据表上加verson版本号字段来表示被修改的次数，当数据被修改时verson加一。线程更新数据时，在读取数据的同时也读取verson，提交更新结果时对比verson值，相等则更新，不等则重试更新操作（重新获取当前数据及verson值，再度更新）直到更新成功。

乐观锁实现方式二：CAS算法

CAS算法思想是 compare and swap（比较与交换），是一种无锁算法（无锁编程：不使用锁的情况下实现多线程间的变量同步，因为是在没有线程被阻塞的情况下实现变量的同步，所以也叫非阻塞同步）。

  • 需要读写的内存值V（原本要改变的值）
  • 进行比较的值A（线程中原值的副本）
  • 拟写入的新值B（线程中更新后的值）

当且仅当V == A时，CAS通过原子方式用新值B来更新V的值，否则不会执行任何操作（比较和替换也是一个原子操作）。一般情况下是一个自旋操作，即不断地重试。

乐观锁缺点

ABA问题

A-B-A，你的女朋友-你出差和别人好上了-你回来后她又悄悄回到你身边了，此时你的女朋友并未改变，还是她。

你的线程读取内存数据a时，a值为8，在你的线程执行完之前，内存中的a被张三线程改为了6，又被李四线程改为了8。此时你的线程执行完毕，发现内存中a还是8，但是….其实它变了。

JDK 1.5 以后的 AtomicStampedReference 类就提供了此种能力，其中的 compareAndSet 方法就是首先检查当前引用是否等于预期引用，并且当前标志是否等于预期标志，如果全部相等，则以原子方式将该引用和该标志的值设置为给定的更新值。解决方式无外乎还是加了一个类似verson的标识。

循环过久开销大

自旋CAS（也就是不成功就一直循环执行直到成功）如果长时间不成功，会给CPU带来非常大的执行开销。

只能保证一个共享变量的原子操作

CAS 只对单个共享变量有效，当操作涉及跨多个共享变量时 CAS 无效。但是从 JDK 1.5开始，提供了AtomicReference类来保证引用对象之间的原子性，可以把多个变量放在一个对象里来进行 CAS 操作。所以我们可以使用锁或者利用AtomicReference类把多个共享变量合并成一个共享变量来操作。

读写锁

共享锁（读锁）

给资源加上读锁后便只可独不可修改，其他线程对此资源也是可读可加锁，但是只能加读锁不能加写锁（多读）

独享锁（写锁）

给资源加上写锁后，线程可以获取并修改资源，其他线程则无法获取资源（单写）

公平锁&不公平锁

公平锁

争抢锁的顺序按照先来后到，新进线程，直接进入wait的等待队列

不公平锁

新进线程，直接去申请锁（由线程调度器如何决定锁的分配）

可重入锁&不可重入锁

可重入锁

synchronized（隐式锁）、ReentrantLock（显式锁）

锁里面可以套锁，线程拿到一把锁之后，可以自由进入同一把锁所同步的其他代码，比如一个方法中的synchronized块中再次调用该方法，再次进入synchronized块，lock也是如此，同一个线程lock获取锁后，可以再次获取，并且lock提供有获取锁的次数的方法，同时释放时，也要释放相应的次数，多次调用unlock

不可重入锁

与可重入锁相反

自旋锁&适应性自旋锁

自旋锁

属于乐观锁。循环的使用cas对数据尝试更新，直到成功

适应性自旋锁

如果锁被占用的时间很短，自旋的效果就很好。反之，如果所被占用的时间很长，自旋就是在白白浪费处理器时间。所以，自旋等待的时间必须要有限度，默认情况下是10次，也可以通过-Xx:PreBloackSpin来更改。如果在自旋10次都没有获得锁，就应该挂起线程。
自旋锁在Java1.6中改为默认开启，并引入了自适应的自旋锁。自适应意味着自旋的次数不在固定，而是由前一次在同一个锁上的自旋时间和锁的拥有者的状态共同决定。如果在同一个锁对象上，自旋等待刚刚成功获得过锁，并且持有锁的线程正在运行中，那么虚拟机就会认为这次自旋也是很可能再次成功的，进而它将会允许线程自旋相对更长的时间。如果对于某个锁，线程很少成功获得过，则会相应减少自旋的时间甚至直接进入阻塞的状态，避免浪费处理器资源。

Future

jdk5新增接口，提供了一种异步并行计算的功能，若主线程遇到一个很耗时的计算任务，可以通过future把这个任务放到异步线程中去解决，主线程继续去处理其他任务。

异步多线程任务执行且要有返回结果（要求：多线程、异步、有返回结果）

FutureTask

实现了Runnable、Future接口，满足多线程和异步要求

又提供有可注入Callable的构造方法，满足有返回值的要求

future+线程池异步多线程任务配合，能显著提高程序执行效率

get()阻塞：在主线程中访问异步线程的返回值（FutureTask.get()），此时主线程会等待异步线程执行完毕获取返回值后再往下执行。一般建议放在程序后面。假如不愿等待很久，希望过时不候，可以自动离开（FutureTask.get(long timeout, TimeUnit unit)）。

isDone()轮询：轮询容易导致CPU空转，耗费更多系统资源。

Future对于结果的获取不是很友好，只能通过阻塞或者轮询的方式得到任务的结果

想将多个异步任务的计算结果组合起来，后一个异步任务的计算结果需要前一个异步任务的值，将两个或多个异步计算合成一个异步计算，这几个异步计算互相独立，同时后面这个又依赖前一个处理的结果

要求：回调通知、创建异步任务、多个任务前后依赖可以组合处理、对计算速度选最快
